The Eduba frame.

Computational orchestration is the working name for what Eduba does. In a given client problem, the work usually divides this way:

60%
Traditional code and database operations. SQL, schema, services, plain engineering.
30%
Rule-based logic. Deterministic workflow, policy, routing.
10%
Genuine AI. Work where a model is the right tool, not the reflex tool.

Teams who skip that separation spend budget putting an LLM where a SQL query would do the job faster, cheaper, and with better guarantees. For a services firm, the same discipline applies one level up. On every new AI agent engagement, the architectural question is which pieces belong in code, which pieces belong in an interpretable context layer, and which pieces belong in the model itself. Getting that separation right once, publishing it as a methodology, and running it consistently across accounts is what protects premium pricing against Copilot-compressed hours.

The case.

Reference engagement

Feeld

Feeld's CTO brought Eduba in for a scoped sprint. The deliverables were a workshop, advisory calls, and an Organizational Context Architecture that defined how the product team's AI workflow would operate at scale. Interpretable Context Methodology (ICM) was the architectural spine.

What Feeld ended up with was a shared vocabulary and a reproducible structure for how context gets written, versioned, and audited inside their agent stack. The same pattern applies to any services firm delivering agents for clients and wanting the architecture to be consistent, auditable, and reusable across accounts.

The paper.

The methodology is published. Interpretable Context Methodology, submitted to ACM TiiS. Folder structure as agent architecture. L0 through L4, with measurable interpretability gains and a 52-member practitioner community testing the edit patterns.
